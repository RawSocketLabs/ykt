🔑 ykt

Getting started — an offline CA in your pocket

ykt lets a pair of YubiKeys act as a certificate authority for SSH and mTLS: the private keys are born on the hardware and never leave, you sign short-lived certificates at a brief touch-gated session, and every machine trusts one signer instead of a pile of copied keys.

Reference: OverviewInstallRecoverySecurity

01 · why this exists

The problem: SSH access sprawls

The default way to grant SSH access is to copy a public key into every server's authorized_keys. That works until it doesn't: keys pile up on every host, there's no expiry, rotating or removing one means editing every machine, and a leaked private key is a standing liability with no clean off-switch. Passwords are worse.

Fig. 1 — Certificates invert the model: hosts trust the signer, not a list of keys.

ykt's answer: a small, offline certificate authority living on YubiKeys. You sign short-lived certificates; hosts trust the CA once; the CA's private key never leaves the hardware. Add a person, rotate a key, or revoke access without touching a single server.

02 · how it's built

The model — four moving parts

Everything is one of four things. Keep them straight and the commands make sense.

anchor — the offline CA YubiKey daily key — your everyday YubiKey certificate — public, signed, short-lived

Fig. 2 — Anchors sign; your one daily key carries a cert per domain; hosts trust the domain's CA.

the security core

The CA's private keys are generated on the anchor YubiKey and can never be extracted. A stolen laptop leaks nothing — the secret was never on it. Revocation is short lifetimes plus a revocation list you push to hosts.

03 · the whole chain

From zero to logging in

This is the full sequence, start to finish. Steps 1–2 happen once per anchor; 3–5 once per person; 6 once per host. The tag on each step says where it runs.

Fig. 3 — The setup chain. Colour = which key/role: blue anchor, brass daily key, teal certificate.

  1. ykt doctor any machine
    Check PC/SC, YubiKeys, tools, and USB interfaces. Run ykt doctor --fix to install what's missing.
  2. ykt init ca a1 anchor holder
    Genesis the CA on an anchor YubiKey (factory-resets its PIV, generates the CA keys on-device). Once per anchor.
  3. ykt init user alice work your machine + your daily key
    Provision your everyday YubiKey and queue a signing request for the work domain.
  4. ykt cert sign a1 anchor holder
    Insert the anchor, review the queue, touch to sign. Produces your certificate(s).
  5. ykt cert install alice work your machine
    Place the signed cert into ~/.ssh (and stash it on the key so a new machine needs no files — see setup key).
  6. sudo ykt init host work on each server — or ykt remote install work --apply from your machine
    Make hosts trust the work CA.
  7. ykt setup ssh init then ssh web1 your machine
    Wire ~/.ssh/config and connect — no password, one touch.
not sure what's next?

Run ykt flow — it assesses the whole system, lists the pending steps in order, waits for the right YubiKey, and runs each one with your approval. It's the guided version of the chain above.

04 · in practice

A few worked examples

① Get yourself logging into a host

# on your machine (daily key inserted)
ykt init user alice work        # enroll + queue a request
# — hand off to the anchor holder —
ykt cert sign a1                 # sign the queue (anchor inserted)
# back on your machine
ykt cert install alice work     # place the cert in ~/.ssh
ykt setup ssh init && ssh web1  # wire config + connect

② Onboard a new server

ykt data inventory add web2 --domain work --address 203.0.113.9
ykt remote collect work web2    # fetch + pin its host key
ykt cert sign a1                 # sign the host cert
ykt remote install work web2 --apply   # push trust over SSH

③ New laptop — carry only the key

# nothing to copy: the cert + key stub ride on the YubiKey
ykt setup key                    # reads them off the inserted key, writes ~/.ssh
ssh web1                        # done

④ Rotate or revoke

ykt cert expiring                # what's due to lapse soon
ykt init user alice work --keep  # re-queue without resetting the key
ykt cert revoke work 1004        # revoke a serial → KRL
ykt remote install work --apply  # push the revocation everywhere
